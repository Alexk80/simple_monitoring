# ML Monitoring System

## Описание проекта

Проект представляет собой приложение имеющее микросервисную архитектуру. Приложение генерирует, обрабатывает и логирует потоки данных посредством возможностей брокера rabbitMQ.

Система состоит из нескольких микросервисов, которые обмениваются сообщениями через RabbitMQ по протоколу AMQP. Данные поступают в потоковом режиме: один сервис генерирует наблюдения, второй строит предсказания, третий считает ошибку модели и сохраняет метрики, а четвёртый строит визуализацию распределения ошибок.

Основная цель проекта — продемонстрировать потоковую обработку данных, обмен сообщениями между сервисами, сопоставление сообщений по идентификатору и логирование метрик в реальном времени.

## Архитектура системы

В системе используются следующие сервисы:

- `features` — источник данных;
- `model` — сервис предсказаний модели;
- `metric` — сервис расчёта ошибки;
- `plot` — сервис визуализации;
- `rabbitmq` — брокер сообщений.

Схема взаимодействия
В соответствии со схемой взаимодействия, указанной в задаче.

## Назначение сервисов

### 1. `features`
Сервис `features` работает в бесконечном цикле и выполняет следующие действия:

- случайным образом выбирает одно наблюдение из датасета;
- формирует уникальный идентификатор сообщения `id`;
- отправляет в очередь `features` вектор признаков;
- отправляет в очередь `y_true` истинный ответ;
- делает паузу между итерациями.

Таким образом реализуется потоковая генерация данных, и помещение их в очередь.

### 2. `model`
Сервис `model`:

- слушает очередь `features`;
- принимает сообщения с признаками;
- формирует предсказание `y_pred`(добавляет шум в линейный регрессор);
- отправляет результат в очередь `y_pred`.

### 3. `metric`
Сервис `metric`:

- читает истинные ответы из очереди `y_true`;
- читает предсказания из очереди `y_pred`;
- сопоставляет сообщения по идентификатору `id`;
- вычисляет абсолютную ошибку: absolute_error = abs(y_true - y_pred)
- записывает результат в файл `logs/metric_log.csv`.

Так как сообщения приходят асинхронно из разных очередей, сервис использует временный буфер в памяти.

---

### 4. `plot`
Сервис `plot` работает в бесконечном цикле и:

- читает файл `logs/metric_log.csv`;
- извлекает столбец `absolute_error`;
- строит гистограмму распределения ошибок;
- сохраняет график в `logs/error_distribution.png`.

График обновляется по мере появления новых строк в CSV-файле.

---

## Формат сообщений

Все сообщения передаются через RabbitMQ в формате JSON-словаря.

Пример сообщения:
{
“id”: “1712345678.123456”,
“body”: 295.0
}

### Пример сообщения с признаками
{
“id”: “1712345678.123456”,
“body”: [0.038, 0.051, 0.062, …]
}

### Пример сообщения с истинным ответом
{
“id”: “1712345678.123456”,
“body”: 295.0
}

Один и тот же идентификатор используется для признаков, истинного ответа и предсказания. Это необходимо для корректного сопоставления данных в сервисе `metric`.

## Используемый датасет

Используется функция генератор, которая случайным образом формирует 3 пчисловых ризнака (features) и шум, затем вычисляет таргет по формуле: y_true = 2 * features[0] + 3 * features[1] + 0.5 * features[2] + random.uniform(-10, 10)

## Логика работы системы

### Генерация потока данных
Сервис `features` на каждой итерации:
1. выбирает случайную строку из датасета;
2. создаёт `id` на основе текущего времени;
3. отправляет признаки в очередь `features`;
4. отправляет истинный ответ в очередь `y_true`;
5. делает задержку `sleep(10)`.


### Построение предсказания
Сервис `model`:
1. получает признаки из очереди `features`;
3. вычисляет предсказание по формуле, как в features: y_pred = 2 * features[0] + 3 * features[1] + 0.5 * features[2] + random.uniform(-10, 10);
4. отправляет сообщение с тем же `id` в очередь `y_pred`.

---

### Сопоставление `y_true` и `y_pred`
Поскольку сообщения из очередей `y_true` и `y_pred` приходят независимо друг от друга, сервис `metric` использует промежуточный буфер.

Пример содержимого буфера:
buffer = {
“1712345678.123456”: {“y_true”: 295.0, “y_pred”: None},
“1712345688.654321”: {“y_true”: None, “y_pred”: 118.44}
}

Когда для одного `id` появляются оба значения:
- вычисляется абсолютная ошибка;
- строка добавляется в CSV;
- запись удаляется из буфера.

### Построение графика
Сервис `plot` периодически перечитывает CSV-лог и обновляет график распределения ошибок.

Это позволяет видеть изменение качества модели по мере поступления новых данных.

После запуска проекта должны выполняться следующие условия:

- все сервисы запускаются через `docker compose`;

- сообщения корректно сопоставляются по идентификатору `id`.


## Особенности реализации

### Бесконечный поток
Поток данных реализован через `while True` в сервисе `features`.

### Уникальный идентификатор сообщения
Для генерации `id` используется текущее время: message_id = datetime.timestamp(datetime.now())

### Работа с асинхронными сообщениями
Так как `y_true` и `y_pred` поступают из разных очередей и в разное время, для их сопоставления используется словарь-буфер.

### Хранение результатов
Результаты вычисления метрик сохраняются в CSV-файл в директории `logs`, которая примонтирована в контейнеры через volume.

### Визуализация
Сервис `plot` строит гистограмму ошибок и сохраняет её как PNG-файл, доступный на хост-машине.
